## 三\. T1\-2\-1：NineToothed 代码生成特化增强挑战

### 3\.1 问题背景

NineToothed 当前代码生成以正确性和通用性优先，已经包含部分特化逻辑，例如根据 shape、stride、contiguity、AOT variant 等信息生成不同代码路径。但在真实算子场景中，仍可能存在一些输入形态已经可以被可靠识别，却仍落入较通用代码生成路径的情况，导致生成的 Triton 源码包含冗余 mask、冗余 stride/pointer arithmetic、冗余广播处理，或没有命中更高效的 contiguous/divisible/layout\-known 路径。

本题要求参赛队在赛题组限定的特化类别中选择 1\-2 类，完成从弱势场景识别、特化条件设计、代码生成修改、fallback 保证到 benchmark 证明的完整闭环。评审重点不是“提交了多少个优化点”，而是特化条件是否可靠、代码生成是否更高效、未命中特化时是否仍保持通用路径正确。

### 3\.2 允许的特化类别

参赛队必须从以下类别中选择 1\-2 类实现，不接受超出范围的自由发挥作为主要评分点：

1. `contiguous fast path`：当输入/输出 tensor 的 contiguous、offset、stride 信息足以可靠判定线性访问时，生成更简洁的线性 pointer 表达式，减少多维 stride/offset 展开。

2. `divisible tile fast path`：当 AOT variant 或静态 shape 信息足以判定 tile 覆盖无尾块时，生成无边界 mask 或更少 mask 的 `tl.load` / `tl.store` 路径。

3. `broadcast / scalar fast path`：当广播维度、size\-1 维度或标量输入可以可靠判定时，简化对应 load、offset 和 mask 表达式。

4. `layout-known AOT variant`：利用 AOT 已知的 contiguity、divisibility、stride 或 shape 约束，选择或生成更专门的 kernel variant，避免可判定场景落入过度通用路径。

特化可以组合，但不得通过识别测试文件名、benchmark 名称、函数名或固定输入尺寸来命中特化。

### 3\.3 固定任务

参赛队必须完成以下任务：

1. 提交 weakness analysis，指出 baseline 在至少 2 个具体 case 下的弱势代码生成表现，并说明低效点属于以下哪类：

    - 冗余 mask。

    - 冗余 stride 或 offset 计算。

    - 冗余 pointer arithmetic。

    - 可判定布局没有命中特化 variant。

    - 广播、标量或 size\-1 维度没有被简化。

2. 在 `src/ninetoothed/generation.py` 或 `src/ninetoothed/aot.py` 中实现 1\-2 类受约束特化。

3. 为每个特化提供明确的启用条件，并保证条件不满足时回到原通用路径。

1. 保持现有测试语义不变，不得删除、跳过或弱化既有测试。

2. 新增 generated source 结构测试，证明特化路径确实被触发，并证明 fallback 路径仍存在。

3. 新增 benchmark，至少覆盖 2 个命中特化的 case 和 2 个不应命中特化的 fallback case。

### 3\.4 必须交付

PR 必须包含：

1. 代码改动：主要位于 `generation.py` / `aot.py`，必要时可在 `tensor.py` 中增加小范围元信息查询或 helper。

2. 测试：

    - 至少 2 个 specialization hit 测试。

    - 至少 2 个 fallback correctness 测试。

    - 至少 2 个 generated source 结构测试，检查 mask、stride/pointer 表达式或 variant 命中情况。

3. benchmark：

    - 输出 JSON 或 CSV。

    - 每个场景至少记录 `baseline_runtime_ms`、`submitted_runtime_ms`、`speedup`、`specialization_hit`。

    - 至少记录一个 generated code 指标：`mask_expr_count`、`stride_expr_count`、`pointer_expr_count`、`variant_name` 或 `source_line_count`。

4. 报告：

    - weakness analysis。

    - 选择的特化类别和启用条件。

    - fallback 条件和正确性证明。

    - 指标对比、性能回退和未覆盖场景。

### 3\.5 隐藏评测

隐藏评测包括：

1. `T1-2-1-hidden-correctness`：30 个 correctness 用例，覆盖 specialization hit 与 fallback。

2. `T1-2-1-hidden-specialization-coverage`：12 个特化命中判定用例。

3. `T1-2-1-hidden-generated-code-metric`：8 个 generated source 结构用例。

4. `T1-2-1-hidden-benchmark`：8 个 runtime benchmark 场景。

隐藏 generated code metric 会按赛题组脚本统计以下指标之一或多个：

- `tl.load` / `tl.store` mask 使用数量。

- stride / offset 表达式数量。

- pointer arithmetic 表达式数量。

- 是否命中指定 AOT variant。

- generated source line count。

### 3\.6 评分标准

总分 100 分。正确性门槛：隐藏 correctness 必须至少通过 29/30；未过门槛的提交总分最高 40 分。

|维度|分值|量化方式|
|---|---|---|
|Correctness|30|隐藏 correctness 30 例，按 `30 * passed / 30` 计分|
|Specialization Coverage|20|隐藏特化命中 12 例，按有效命中比例计分；误命中 fallback case 扣分|
|Generated Code Metric|20|隐藏 generated source 8 例，按 mask、stride/pointer 或 variant 指标改善比例计分|
|Runtime|20|隐藏 benchmark 8 例，按 speedup 线性计分|
|工程与报告质量|10|weakness analysis、fallback 设计、代码边界、测试质量、无硬编码和引用披露|

Specialization Coverage 计分规则：

- 每个应命中特化的用例正确命中得 1 分。

- 每个应回退的用例错误命中特化扣 1 分。

- 覆盖率分数按 `max(0, hit_score - false_hit_penalty) / total_cases` 折算到 20 分。

Generated Code Metric 计分公式：

- 每个用例计算 `reduction = (baseline_count - submitted_count) / baseline_count`，或按 variant 命中布尔指标计分。

- `reduction >= 0.25` 得满分。

- `0 < reduction < 0.25` 按 `reduction / 0.25` 线性计分。

- `reduction <= 0` 得 0 分。

- 8 个用例取平均后折算到 20 分。

Runtime 计分公式：

- 每个 benchmark 计算 `speedup = baseline_runtime / submitted_runtime`。

- `speedup >= 1.10` 得满分。

- `1.00 <= speedup < 1.10` 按 `(speedup - 1.00) / 0.10` 线性计分。

- `0.95 <= speedup < 1.00` 得 30% 分。

- `speedup < 0.95` 得 0 分，并记录一次性能回退。

- 8 个 benchmark 取平均后折算到 20 分。
